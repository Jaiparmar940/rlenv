# V0.2 — planned work

v0.1 is three hand-authored scenarios (one battery fault, two ground faults) on
a single vehicle topology, run against five models. This document states what
v0.2 changes and why. It is a plan, not a record of work done; nothing here is
implemented on `main`.

## 1. Hard tier

The v0.1 scenario set is cleared by frontier models (mean 94.6, full root-cause
credit in every episode). It no longer discriminates at the top. The hard tier
adds difficulty along axes that v0.1 deliberately excludes:

**Intermittent faults.** A fault that manifests on only some attempts. The
scoring consequence is the point: one clean reading must not exonerate the
component, so an agent that probes once and moves on should be wrong, and an
agent that re-probes should be able to establish the pattern. Intermittency
must be *deterministic per seed* — drawn from the scenario's seeded RNG, not
from wall-clock time or unseeded randomness — or the environment stops being
reproducible and scores stop being comparable across runs.

**Compound faults.** Two simultaneous faults, a primary and a secondary. The
grader accepts both: a diagnosis naming only one is incomplete, and the
resolution check has to require that the vehicle actually starts, which it will
not until both are addressed. This is distinct from the v0.1 red herring, which
is a *bait* (an innocent component that reads tempting) rather than a second
real fault.

**Cost pressure.** Tighter expert baselines, so the cost-efficiency bucket bites
earlier, and a **price-weighted wrong-part penalty**. This closes a known gap in
v0.1: the wrong-part penalty is a flat −8 whether the part is a $15 fusible link
or a $450 ECU. Parts dollars therefore influence *no* score bucket today (the
cost bucket became time-only in v0.1 to stop parts spend from diluting a time
overrun). Weighting the penalty by part price puts parts dollars back into the
score, on the axis where they belong.

**A larger action space.** More tools and more parts means more ways to be
wrong, and a lower prior on stumbling into the right replacement.

**pass^k as the headline metric, not the mean.** A model that is right two times
in three is not deployable, and a mean does not say which model that is. v0.1
already reports pass^k (full root-cause credit in *every* epoch) per model x
scenario; v0.2 promotes it to the headline number and demotes the mean to
supporting detail.

A hard-tier preview exists on the un-merged branch `hard-tier-preview`. It is
**pending human physics sign-off** and is not merged, not published, and not
part of the v0.1 numbers.

## 2. Generative architecture

The v0.1 scenarios are hand-authored — a human wrote each one and a human
verified each one. That does not scale: verification cost grows linearly with
scenario count, and the verifier is the bottleneck. v0.2 moves authoring behind
a schema.

**(a) Schema-constrained authoring.** A scenario becomes a *declaration*:
topology, which resistance or source parameter a fault perturbs, and the tells
it is expected to produce. Never hand-written observations. v0.1 already holds
this line by convention (`Never hand-author voltages` — change R or I and let
V = I x R produce the reading); the schema makes it structurally impossible to
break rather than a rule to remember.

**(b) An assumption ledger.** Every physical constant carries a provenance
state: `measured` / `sourced` / `estimated` / `UNVERIFIED`. The repo today marks
unverified constants with inline `# TODO(VERIFY)` comments — the charging-system
constants added for the `running` state are marked exactly this way. That works
until there are a hundred of them, at which point "which constants has a human
actually signed off on?" has no answer short of grepping. A ledger makes the
unverified set *enumerable and queryable*, and lets the eval refuse to publish a
scenario that depends on an `UNVERIFIED` constant.

**(c) An invariant engine.** The physics invariants — load sag never inverted,
resting monotonicity down the supply path, red herrings resting-only, the fault
uniquely localizable — become executable checks that run against **every
generated scenario**, not per-scenario tests written by hand. Each of these
invariants exists because a human caught a bug that the automated test suite had
passed; encoding them as universal checks is what stops that class of bug from
returning at scale.

**(d) A sampler.** Given the causal model, generate scenario *families*: the
same fault class across different components, magnitudes, and vehicle
topologies.

The key property is where validation attaches. It attaches to the **causal
model**, not to individual scenarios. Verify the model once — the constants, the
propagation rules, the invariants — and every scenario the sampler generates
inherits the guarantee. That is what makes human verification scale sublinearly
with scenario count, and it is the difference between a benchmark that ships
three scenarios and one that ships three hundred.

## 3. The scale curve as the standing results format

`results/scale_curve.md` (generated by `scripts/scale_curve.py`) is the standing
output of a run, not a one-off analysis: every model tier, per-scenario
reliability, pass^k, verified-fix rate, and time-vs-expert. Any change to the
environment or the grader is re-reported in that format.

The v0.1 run leaves one row unmeasured: the **open 3B–8B tier** — the weight
class that would actually run on a robot — has no numbers, because no
OpenRouter key was configured. Those rows read `NEEDS_API_KEY`. Closing that row
is the first v0.2 action.

## 4. Closing the small-model gap

The v0.1 finding:

| tier | model | mean | pass^k |
|---|---|---:|---|
| frontier | 4 models | 91–98 | PASS (all) |
| deployment (small closed) | `anthropic/claude-haiku-4-5` | 47.9 | fail |

The gap is not uniform. The deployment-tier model scores 89.8 on the
single-fault battery scenario and 50.0 / 3.9 on the two ground-fault scenarios —
it collapses specifically where a plausible-but-wrong reading has to be
rejected, not on physical diagnosis in general. Its failure mode there is a
parts cannon: across the six ground-fault episodes it replaced eleven innocent
parts (battery 5x, alternator 4x, starter motor 2x) on a car whose fault is a
$25 ground strap.

The open question, stated plainly: **is that gap closable by training on
environments like this one?** Two possibilities, and v0.1 does not distinguish
them:

- It is a *knowledge/procedure* gap — the model does not know to reference its
  measurements to the right ground, does not know that one clean reading is not
  an exoneration, does not have the isolate-repair-verify loop. If so, it is
  trainable, and this environment is training data.
- It needs *capability* the small model does not have — holding a multi-step
  causal hypothesis across a dozen tool calls while resisting a salient wrong
  answer. If so, no amount of training on this environment closes it, and the
  finding is a ceiling, not a curriculum.

An environment that separates two adjacent tiers of the same vendor by 56 points
is the instrument for answering that either way. That is the thesis, and it is a
question, not a claim. v0.2 exists to answer it.

## 5. Next domain: robotics-cell fault triage

A robotic work cell stops. The agent has to localize the fault across layers
that look nothing alike from the outside but present identically at the symptom
— a mechanical bind, a blown drive fuse, a drifted or dead sensor, a controller
fault — using the tools a technician actually has: fault codes, joint-current
and encoder telemetry, physical inspection, and part swaps that cost money and
downtime. Same architecture as this repo (a causal model, tool outputs that are
instrument readings only, a grader that rewards localization rather than symptom
relief), different physics.
